# 30. ドキュメント管理

## ディレクトリ構造

```
docs/
├── prd/         # Product Requirements Document
├── adr/         # Architecture Decision Record
├── tech/        # 技術文書（DBスキーマ、API設計等）
├── api/         # API仕様書
├── operations/  # 運用文書
└── user/        # ユーザーガイド
```

## ドキュメントタイプ別ルール

### PRD（Product Requirements Document）

**配置**: `docs/prd/<機能域>-<内容>.md`（例: `auth-requirements.md`）

```markdown
# [機能名] 要件定義

## 概要
目的と背景

## 機能要件
- 必須機能 / 推奨機能

## 非機能要件
- パフォーマンス / セキュリティ

## 受け入れ基準
- シナリオごとの 条件 / 実行 / 結果
```

### ADR（Architecture Decision Record）

**配置**: `docs/adr/<3桁連番>-<決定内容>.md`（例: `001-initial-architecture.md`）
**変更ポリシー**: 既存ADRは変更せず、新ADRで上書き決定を記録

```markdown
# ADR-001: <決定タイトル>

## ステータス
採用済み / 廃止 / 検討中

## 背景
なぜこの決定が必要だったか

## 検討した選択肢
### 選択肢1
- メリット / デメリット

## 決定
採用した内容

## 理由
判断の根拠

## 結果
影響と将来の見通し
```

### 技術文書

**配置**: `docs/tech/<分野>-<内容>.md`（例: `database-schema.md`）

### API仕様書

**配置**: `docs/api/`
**形式**: エンドポイントごとに リクエスト / レスポンス / エラーレスポンス を JSON で例示

## 更新頻度

| 種別 | 更新タイミング |
|---|---|
| PRD | 要件変更時に即座 |
| ADR | 決定時に作成（既存は不変） |
| 技術文書 | 実装変更時に同期 |
| API仕様書 | API変更時に即座 |

## 品質基準

- 目的と対象読者が明確
- 実装と乖離していない
- 日本語で記述
- 最終更新日を記載

## エージェント運用ルール

### 作成時
- 必ず適切な `docs/` 配下に配置
- 新規ドキュメントは明示的要求時のみ作成
- テンプレートに従う

### 参照時
- 正しいパスで参照
- 関連ドキュメントとの整合性を確認

### 更新時
- 影響範囲を特定し関連ドキュメントも併せて更新
- 変更理由を明確化

## 定期メンテナンス

- 月次: ドキュメントと実装の乖離チェック
- 四半期: ドキュメント体系の見直し
- リリース前: 関連ドキュメントの更新確認

## 新規メンバーオンボーディング推奨順

1. `README.md` → プロジェクト概要
2. `docs/prd/` → 要件理解
3. `docs/adr/` → アーキテクチャ決定の経緯
4. `docs/tech/` → 技術詳細
