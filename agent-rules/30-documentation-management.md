# 30. ドキュメント管理

## ディレクトリ構造
- `docs/prd/`: 要件定義 (PRD)
- `docs/adr/`: アーキテクチャ決定レコード (ADR)
- `docs/tech/`: 技術文書（DBスキーマ、API設計等）
- `docs/api/`: API仕様書（JSONでリクエスト/レスポンスを例示）
- `docs/operations/`: 運用文書
- `docs/user/`: ユーザーガイド

## ドキュメント規約

### PRD
- **配置**: `docs/prd/<機能域>-<内容>.md`
- **構成案**: 概要（目的と背景） / 機能・非機能要件 / 受け入れ基準（条件・実行・結果）

### ADR
- **配置**: `docs/adr/<3桁連番>-<決定内容>.md`
- **ポリシー**: 既存ADRは変更せず、新ADRで上書き（ステータス: 採用済み / 廃止 / 検討中）
- **構成案**: 背景 / 検討した選択肢（メリット・デメリット） / 決定 / 理由 / 結果（影響）

### 技術文書
- **配置**: `docs/tech/<分野>-<内容>.md`

## 運用＆ライフサイクル

### 更新タイミング
- PRD / API仕様書: 仕様変更時に即座
- ADR: 技術決定時に新規作成
- 技術文書: 実装変更時に即座に同期

### 品質基準
- 対象読者の明確化、日本語記述、最終更新日の記載。

### 作成原則
- **プランニング時のコードスニペット禁止**: 計画・設計段階の文書では過度なコードスニペットを記載しない。意図・方針・インターフェースを自然言語で記述し、実装の詳細はコード本体に委ねる（コードと文書の二重管理・乖離を防ぐため）。例示が不可欠な場合のみ最小限に留める。
- **アスキーアート禁止**: 図示が必要な場合はアスキーアート（ASCII art）を使わず、Mermaid 等のテキストベース作図記法を用いる（保守性・可読性・差分追跡性のため）。ファイル構成・ディレクトリツリーも罫線（`├──` 等）で描かず、表または箇条書きで表現する。
- **設計書（HTML 含む）も同原則を適用**: `docs/design/` 配下の設計書（`*.html` のビジュアル設計書を含む）も上記 2 原則に従う。実装コード（クラス本体・関数実装・設定ファイル中身等）は掲載せず、責務・インターフェース・データ契約は自然言語・表・Mermaid／SVG で示す。API リクエスト/レスポンスや DB スキーマなど例示が不可欠な場合は、最小限の JSON 例またはフィールド表（項目・型・説明）に留める。実装の正本はコード、設計書は意図と契約の記述に徹する。

### エージェント行動指針
- **作成**: 明示的要求時のみ適切なテンプレートに従い作成。
- **参照**: 正確なパスで参照し、関連ドキュメントとの整合性を検証。
- **更新**: 影響範囲のドキュメントを同時に更新し、変更理由を明記。

### メンテナンス＆オンボーディング
- **確認**: 月次（乖離チェック）、四半期（体系見直し）、リリース前（更新確認）。
- **推奨読順**: `README.md` ➔ `docs/prd/` ➔ `docs/adr/` ➔ `docs/tech/`
