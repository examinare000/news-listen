# 13. 可読性ルール

## 基本方針
「書く時間」より「読まれる時間」の方が圧倒的に長いことを意識し、**読み手にとって分かりやすいコード**を書く。

## 具体的なルール

### 1. ネストを浅く保つ (Early Return)
`if-else` の深いネストは避け、ガード節（Guard Clause）を使って早期リターンする。

```typescript
// Bad
if (user) {
  if (user.isActive) {
    // ... logic
  }
}

// Good
if (!user || !user.isActive) return;
// ... logic
```

### 2. 命名規則
- **変数名**: 内容を具体的に表す名前をつける（`data`や`tmp`は避ける）。
- **関数名**: 動詞から始め、何をするかを明確にする（`getUser`, `calculateTotal`）。
- **一貫性**: プロジェクト内で命名規則（camelCase, snake_case等）を統一する。

### 3. コメント
- **What**（何をしているか）ではなく、**Why**（なぜそうしているか）を書く。
- コード自体が自己説明的であればコメントは不要。

### 4. 関数の長さ
- 1つの関数は1つのことだけを行う（単一責任）。
- 長すぎる関数は、意味のある単位で小さな関数に分割する。
