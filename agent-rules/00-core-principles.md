# 00. 基盤原則

すべてのエージェント・全ファイルに優先する絶対原則。例外なし。

## 絶対遵守の3原則

### 1. デグレッション防止

- 既存の動作している機能を絶対に壊さない
- 新機能追加時は既存テストがすべて通ることを確認
- 動作確認済みの設定や実装は不用意に変更しない
- 問題発生時は速やかに動作していた状態に戻す

### 2. 日本語使用

- ユーザーへの返答・コメント・エラーメッセージ・テスト記述・コミットメッセージ・ドキュメントはすべて日本語
- 思考や検索は英語可
- 変数名・関数名・クラス名は英語の慣習に従う

### 3. Test-Driven Development（t-wada方式）

- Red → Green → Refactor サイクルを厳守
- テストを「生きた仕様書」として機能させる
- カバレッジは副産物。振る舞いに集中する
- 言語に関係なく適用（詳細は `11-testing-strategy.md`）

## 開発サイクルの固定順序

作業開始時に必ずこの順序で進める。順序の入れ替えは禁止。

```
1. フィーチャーブランチを作成（詳細: 10-git-strategy.md）
2. ADR・設計ドキュメントを更新（詳細: 30-documentation-management.md）
3. テストを先に書く（詳細: 11-testing-strategy.md）
4. TDDサイクルで実装
5. アトミックコミット（詳細: 10-git-strategy.md）
6. 品質チェック（test / lint / typecheck）→ マージ
```

## コミット前ユニバーサルチェックリスト

1. すべてのテストが通っているか
2. 型チェックがクリーンか
3. lint違反が修正されているか
4. シークレット・機密データが含まれていないか
5. 各コミットがアトミックで適切に説明されているか
6. 一時的なデバッグコードが削除されているか

## 適用範囲

- 全言語・全フェーズ・全プロジェクトに適用
- 他のすべての指示と矛盾する場合、本原則を優先
- 矛盾解決の最終基準は本ファイル
